\section{Overview}

This document analyses the completeness of our Music History knowledge graph (42,554 triples, 58 primary artists) and describes how RAG was used to resolve identified gaps. Completeness was measured using CM1--CM4 metrics \cite{zaveri2015}, as taught in Week~10.

Gaps were identified through three methods: (1)~CM1--CM4 metrics measuring schema coverage and property completeness, (2)~SPARQL query results where sparse results indicated missing data, and (3)~domain expert review of the TTL file. The team discussed the analysis after Meeting~3 and agreed to apply RAG to all 10 gaps.

\section{Ontology Gaps}

\textbf{O1. \texttt{signedTo} lacks temporal scoping.} The property links artists to all their labels simultaneously (229 triples) with no start/end dates. CQ17 produces cartesian products. RAG prompt P34 provided 226 temporal ranges (46\% high confidence), demonstrating feasibility. Implementing temporal scoping requires RDF reification, which was documented as future work.

\textbf{O2. \texttt{performedAt} sparsely populated.} Initially only 5 triples from 3 artists. No structured source provides concert data. RAG prompt P35 identified 113 events across 43 artists. The use of programmatic ingestion and LLM API extraction brought the total to 131 triples.

\textbf{O3. \texttt{collaboratedWith} modelled indirectly.} MusicBrainz models collaborations as band/project entities rather than direct artist links. Starting from 76 triples, RAG prompt P36 identified 69 new direct pairs. Full LLM API extraction (58 artists) raised the total to 242 triples (3.2$\times$ increase).

\textbf{O4. \texttt{founded} barely represented.} Only 4 triples existed. RAG prompt P37 identified 47 new entries (labels, bands, foundations). Final count: 69 triples (17$\times$ increase).

\textbf{O5. \texttt{hasMusicalPeriod} limited.} Only 5 triples linked 3 artists to periods. RAG prompt P38 provided assignments for all 58 artists. Final count: 126 triples, covering every primary artist.

\section{Instance Gaps}

\textbf{I1. Secondary artists lack data.} Of 1,332 artist entities, only 58 are primary. The remaining 1,274 had minimal properties (\texttt{countryOfOrigin} 11.4\%, \texttt{genre} 4.6\%). RAG prompt P39 enriched 50 secondary artists (94\% resolved). The pipeline's enrichment function was also extended to fetch genres, birth/death dates, and type, adding 216 triples for 102 artists.

\textbf{I2. Umm Kulthum wrong entity.} MusicBrainz search matched ``UMM'' (electronic project) instead of the Egyptian singer, causing a cascading failure: wrong entity $\rightarrow$ no Wikidata $\rightarrow$ no Wikipedia $\rightarrow$ no data. RAG prompt P40 provided comprehensive data but \textbf{hallucinated the MBID} (returned 404). The correct MBID was found via programmatic API search. Fully resolved after adding an MBID override. Key insight: LLMs are unreliable for external identifiers; non-Latin scripts cause systemic entity resolution failures.

\textbf{I3. LLM text extraction coverage.} Initially 16/58 artists (298 triples, 28\%). RAG prompt P41 demonstrated the approach for 5 artists, finding 44\% of triples use text-unique predicates. The prompt template was then automated via Google Gemini API. \textbf{Final: 58/58 (100\%), 897 triples.}

\textbf{I4. Hildegard von Bingen missing genre.} The only primary artist without genres. Medieval music predates modern classification, and MusicBrainz, Wikidata, and Discogs all failed. RAG prompt P42 provided 5 genres (medieval music, sacred music, plainchant), 3 periods, and 10 compositions. \textbf{Fully resolved.} Demonstrates RAG compensating for structured source blind spots.

\textbf{I5. Cover composers lack data.} 330 composer entities from cover detection had names/MBIDs but no biographical data. RAG prompt P43 enriched 50 composers (94\% resolved) and surfaced 8 entity duplicates. The enrichment query was broadened to include composers, enabling automatic enrichment.

\section{RAG Strategy and Results}

Each gap was resolved using a 6-step RAG cycle following the ``KG as SPARQL-based retrieval source'' pattern from Week~11: (1)~identify gap from the analysis above, (2)~run a SPARQL query against the KG to retrieve relevant context, (3)~verbalise retrieved triples into natural language, (4)~prompt an LLM with the verbalised context and the gap question, (5)~parse the LLM's JSON response into candidate triples, (6)~ingest into the KG via programmatic entity linking. The ingestion script is integrated into the main pipeline as Step~14 and runs automatically after validation.

\subsection{Concrete Example: Producer Gap-Filling}

To illustrate the cycle, we demonstrate gap-filling for Michael Jackson's producer relationships. The SPARQL retrieval step fetched his 8 albums from the KG, including existing data (Quincy Jones produced Off the Wall, Thriller, and Bad). The RAG prompt provided this context and asked: ``who produced each of these albums?'' The LLM confirmed the 3 existing triples, provided 2 high-confidence new ones (Hal Davis for Got to Be There and Ben), and \textbf{flagged 3 as uncertain} (Dangerous, HIStory, Invincible --- each had multiple co-producers). Only the 2 high-confidence triples were ingested; the 3 uncertain ones were documented for manual validation.

\subsection{From RAG to Automation}

A key outcome was the progression from manual RAG demonstration to pipeline automation. Several RAG findings were subsequently implemented as permanent improvements: I3 to LLM API (text extraction automated via Gemini, raising coverage from 16/58 to 58/58), I1/I5 to enrichment extension (fetch genres, dates, type for secondary artists), and data quality to smart filtering (Wikidata P31/P279* class hierarchy queries replacing manual blacklists).

\section{Completeness Metrics}

CM1 Schema: 93\% to 97\% (29/30 properties populated). CM2 Primary: High to Higher (Country 98\%, Genre 100\%). CM2 All: Low to Improved (+216 triples from enrichment). CM4 Interlinking: Strong to Stronger (Umm Kulthum cross-refs fixed). Text extraction: 28\% to 100\% (897 triples via LLM API). Defined classes: 74 to 191 (+158\%). Total triples: $\sim$38,000 to 42,554 (+12\%).

\section{Reflection}

Three key insights emerged. First, \textbf{RAG is most valuable as a data quality auditor}: the KG vs Plain LLM comparisons (P18--P23) surfaced non-music genres, instrument duplicates, and range violations invisible to automated validation, leading to 6 pipeline fixes. Second, \textbf{LLMs are unreliable for external identifiers}: the Umm Kulthum case showed the LLM hallucinating a plausible MBID that returned 404. Non-Latin scripts cause cascading entity resolution failures. Third, \textbf{automated construction requires manual validation}: ontological correctness (Prot\'{e}g\'{e}) and semantic accuracy (domain expert review identifying 76 non-music genres and 31 non-music awards). The optimal approach is iterative: build $\rightarrow$ validate $\rightarrow$ review $\rightarrow$ fix $\rightarrow$ rebuild.

\subsection{Future Work}
\begin{enumerate}[nosep]
  \item Temporal scoping for \texttt{signedTo} via Wikidata qualifiers or RDF reification.
  \item Concert/venue data from Setlist.fm API or MusicBrainz events.
  \item Multilingual entity resolution to eliminate MBID overrides for non-Latin scripts.
\end{enumerate}
